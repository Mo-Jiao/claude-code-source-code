## 总体评估

教程整体质量较高。核心技术概念（Agent Loop、工具注册、权限系统、上下文压缩、子 Agent 隔离、MCP 协议）的阐述准确且深入，伪代码对源码的还原度令人满意。主要问题集中在三个方面：

1. **归因错误**——"苦涩教训"在不同课程中归因不一致
2. **s16 课程编号交叉引用严重错位**——大量引用了不存在的课程编号
3. **部分行业引用过度概括**——将有限数据点包装为普遍规律

严重性分级：P0（必须修复）2 个、P1（建议修复）5 个、P2（可选优化）6 个。

### 需要修正的概念

#### P0-1: "苦涩教训"归因不一致

**问题：** s01 第 502 行将"苦涩教训"（The Bitter Lesson）归因于 **Boris Cherny**："Claude Code 团队践行 Boris Cherny 提出的'苦涩教训'哲学"。而 s16 第 415 行正确归因于 **Rich Sutton**："Rich Sutton 的'苦涩教训'认为……"

**事实：** "The Bitter Lesson" 是 Rich Sutton 2019 年提出的经典 AI 哲学观点。Boris Cherny 是 Claude Code 团队的工程师，他在博客中引用并实践了这一理念（"每次新模型发布，删掉一半 prompt"），但他不是这个概念的提出者。

**修复建议：** s01 应改为"Claude Code 团队（Boris Cherny）践行 Rich Sutton 提出的'苦涩教训'哲学"或类似措辞，区分概念提出者和实践者。

#### P0-2: s16 课程编号交叉引用全面错位

**问题：** s16 中多处课程编号引用与实际课程内容不匹配：

| s16 中的引用 | 引用内容 | 实际对应课程 |
|-------------|---------|-------------|
| s13 | 系统提示词六层结构、prompt cache | **s03** (system-prompt) |
| s10 | 延迟加载 (Lazy Loading) | **s09** (skills) |
| s14 | 权限级别 (Permission Levels) | **s04** (permissions) |

这些错误出现在 s16 的第 328、374、387、400、468、471-472、481-490、507-508、512、556 行，涉及面极广。推测原因：课程编号在某次重排后未同步更新 s16 的引用。

**修复建议：** 全面搜索 s16 中的 `s10`、`s13`、`s14` 引用并逐一修正。

#### P1-1: "27 种生命周期钩子"数量存疑

**问题：** s05 Key Takeaways 声称"27 种生命周期钩子"（第 5、8 行），源码映射表也提到"27 种事件"（第 846 行）。但课程正文中实际列举的事件数量约为 20 个左右。

**评估：** 如果 27 这个数字来自源码中 `HOOK_EVENTS` 常量的实际长度，则课程正文应补充完整列表或说明省略了哪些。如果 27 是估算值，则应修正为实际数量。

**修复建议：** 核对源码中 `HOOK_EVENTS` 的实际长度，确保课程中的数字与之一致。

#### P1-2: "推理能力在约 3,000 token 处开始衰减"缺乏出处

**问题：** s07 第 902 行声称"推理能力在约 3,000 token 处开始衰减，prompt 的实际甜蜜点在 150-300 词"，引用标记为 [R1-6]。

**评估：** 这个具体数字（3,000 token）作为一个精确的衰减阈值，目前尚未在主流学术文献中被系统验证。不同模型、不同任务类型的衰减曲线差异很大。将其作为确定性事实呈现有误导风险。

**修复建议：** 改为"研究表明推理能力随上下文长度增加而衰减"，或注明这是特定研究/经验观察而非普遍结论。补充 [R1-6] 的具体出处。

#### P1-3: CPU/RAM/OS/App 类比归因需验证

**问题：** s00 将 "LLM=CPU, 上下文窗口=RAM, Harness=OS, Agent=App" 类比归因于 "Philschmid, 2026"。

**评估：** 这个类比在 AI 社区中有多个变体流传（Karpathy 也用过类似比喻）。需要确认 Philschmid 是否确实是最早或最权威的提出者。如果只是其中一个使用者，应注明"类比来源于社区共识，Philschmid 等人使用"。

**修复建议：** 核实归因准确性。如无法确认首创，改为"社区广泛使用的类比（如 Philschmid [R1-1]）"。

### 缺失的细微差别

#### P1-4: "自评估偏差"过度特指化

**位置：** s10 Why 段落

**问题：** 教程将"模型评估自己的输出几乎总是批准"表述为 Anthropic 发现的一个特定现象。实际上这是 LLM 领域的一个已知通用问题（self-consistency bias / sycophancy），并非 Anthropic 独立发现或命名的特定偏差。

**建议：** 改为"LLM 领域已知的自我一致性偏差（self-consistency bias）——模型在评估自己的输出时倾向于给出正面评价"，避免给读者造成这是一个特定研究发现的印象。

#### P1-5: "约束悖论"过度概括

**位置：** s00 第 204 行、s04 第 617 行、s10 第 619 行

**问题：** 教程多次使用"约束悖论"（Constraint Paradox）这一框架：限制 Agent 反而让它更强。这个表述将 LangChain Terminal Bench 的实验结果（改进 harness 提升评分 13.7 分）过度概括为一个"悖论"。

**实际情况：** LangChain 的数据展示的是"更好的工程设计（包含约束）提升了效果"，这不是悖论——这是正常的软件工程实践。真正的洞察是"投入在 harness 层的工程努力比换模型更有效"，而不是"约束本身是悖论性的"。

**建议：** 第一次提出时保留"约束悖论"作为启发性框架，但在后续引用中避免重复使用，改为更精确的表述如"工程约束提升可靠性"。避免让读者误以为这是一个被行业广泛认可的命名概念。

#### P2-1: Explore Agent 省略 CLAUDE.md 的节省量

**位置：** s12 设计决策

**问题：** 提到 "Explore agent 每周被调用 3400 万次以上" 和 "节省了约 5-15 Gtok/周"。这些数字如果来自源码注释则应标注出处，如果是推算则应注明。

## 审查方法说明

本审查基于以下方法：
1. 逐课阅读全部 17 课内容（s00-s16）
2. 核对关键概念的原始出处（Rich Sutton Bitter Lesson、LangChain Context Engineering、Anthropic Building Effective Agents）
3. 检查课程间交叉引用的一致性
4. 对行业引用数据的合理性进行判断
5. 评估 Why 段落的论证质量（论点→论据→结论的逻辑链完整性）

未覆盖：源码路径准确性验证和伪代码与实际代码的对比（由独立审查任务覆盖）。
